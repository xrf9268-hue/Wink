# Localization

Wink ships a String Catalog (`.xcstrings`) with English as the default localization and zh-Hans (Simplified Chinese) as the first translated locale.

## Architecture

- **Source of truth**: `Sources/Wink/Resources/Localizable.xcstrings` — hand-authored JSON, editable in Xcode's String Catalog UI. Every entry uses **English-as-key**: the key is the literal English source string (or its `%@`/`%lld` format-specifier form for interpolated text), so a missed or new-but-untranslated string still renders sensible English instead of a raw key.
- **Compiled output**: `Sources/Wink/Resources/Localized/{en,zh-Hans}.lproj/Localizable.strings` (+ `.stringsdict` for plural entries) — generated by `scripts/gen-localizations.sh` via `xcrun xcstringstool compile`, and **checked into git**.
- **Why the compiled output is checked in**: `swift build` (plain SwiftPM, not an Xcode build) does not compile `.xcstrings` files at all — that auto-extraction/compilation step is an Xcode-build-system feature. Left alone, SwiftPM just copies the raw `.xcstrings` into the resource bundle as inert JSON; even the English strings would not resolve, and every lookup would fall back to the literal key text. Checking in the compiled `.lproj` output is what makes `swift build`/`swift test` correct on any machine — including one without Xcode.
- **Package.swift**: declares `defaultLocalization: "en"`, excludes `Resources/Localizable.xcstrings` from the resource copy (it's source, not a runtime asset), and processes `Resources/Localized` as a resource so the compiled `.lproj` folders land in the built `Wink_Wink.bundle`.
- **Lookup**: all localized lookups go through `WinkResourceBundle.bundle` (`Sources/Wink/UI/MenuBar/WinkResourceBundle.swift`), which already resolves the right bundle in packaged-app, `swift test`, and local-dev-build contexts.
  - SwiftUI: `Text("Ready", bundle: WinkResourceBundle.bundle)`
  - Everywhere else: `String(localized: "Ready", bundle: WinkResourceBundle.bundle)`
  - Many SwiftUI initializers (`Button`, `Label`, `Picker`, `TextField`, `.help`, `.accessibilityLabel`, …) also accept a plain `String` via a `StringProtocol`-generic overload, rendered **verbatim** (no catalog lookup). That's the correct choice when the value is already a localized `String` built elsewhere (e.g. a computed property) — do not re-wrap it in another lookup.
  - `Link` has no `String`/bundle-taking initializer; use the `Link(destination:) { Text("…", bundle: WinkResourceBundle.bundle) }` label-closure form.
  - A few explainer strings (e.g. `GeneralTabView`'s Hyper Key row: "Hold" / `⌃⌥⇧⌘` keycap / "to act as" / `Caps Lock` keycap / ". Tap alone…") compose a sentence from separately-localized fragments around inline keycap glyphs, rather than one flat catalog entry, because the keycap glyphs themselves must stay symbolic. This works for English and zh-Hans (both read left-to-right with the same fragment order), but is not a general-purpose pattern — a locale with different word order or RTL text would need the surrounding sentence restructured, not just each fragment translated. Revisit this composition before adding such a locale.

### The SPM lowercased-lproj quirk

SwiftPM lowercases `.lproj` directory names when it builds the resource bundle, so `zh-Hans.lproj` becomes `zh-hans.lproj` on disk. This is harmless: Foundation's locale matcher (`Bundle.preferredLocalizations`, `Bundle.path(forResource:forLocalization:)`) tolerates the case difference and both `zh-Hans` and `zh-CN` system-language preferences still resolve to it correctly. Don't try to "fix" the casing — `Tests/WinkTests/LocalizationTests.swift` asserts the tolerant behavior directly.

## How to add a new string

1. Write the SwiftUI/Swift call site using the English text as the key (see the bundle patterns above).
2. Add an entry to `Localizable.xcstrings` for that exact key: an `en` `stringUnit` (usually identical to the key) and a `zh-Hans` `stringUnit` with the translation. For interpolated text, match the format specifier exactly — `%@` for `String`, `%lld` for `Int` (64-bit `Int` interpolation in `LocalizedStringKey`/`String.LocalizationValue` always produces `%lld`, not `%d`).
3. For text that genuinely varies by grammatical number in English (e.g. "1 activation" vs "5 activations"), use an xcstrings **plural** `variations` block (`one`/`other`) instead of a single `stringUnit`. zh-Hans has no plural forms, so it stays a single `stringUnit` covering every count.
4. Run `bash scripts/gen-localizations.sh` and commit the regenerated files under `Sources/Wink/Resources/Localized/`.
5. `bash scripts/gen-localizations.sh --check` (also run in CI) fails the build if the checked-in output drifts from the `.xcstrings` source.

## How to add a new locale

1. Add the locale code to `CFBundleLocalizations` in `Sources/Wink/Resources/Info.plist` (this is what makes macOS offer Wink under **System Settings → General → Language & Region → per-app language** — the `.lproj` folders alone are nested inside the SPM resource bundle and invisible to System Settings otherwise).
2. Add a `localizations` entry for the new locale to every string in `Localizable.xcstrings` (Xcode's String Catalog editor can do this in bulk).
3. Regenerate and commit as above.

## Gotchas

- Never localize: debug/log strings, `UserDefaults`/persistence keys, notification names, the `wink://` URL scheme, bundle identifiers, file names, keycap glyphs (`⌃⌥⇧⌘`, `F19`, `Caps Lock`, `Hyper`), the "Wink" brand name, or app display names sourced from `NSWorkspace`.
- Never switch/compare on a value that has been through `String(localized:)` — that breaks the moment the string stops being English. If a string genuinely needs to be both a display value and a decision input (see `InsightsChange` for a fixed example), carry a separate non-localized enum for the decision and keep `text` display-only.
- Never persist a localized value either. `AppShortcut.frontmostTargetStableName` ("Current App", a plain literal) is what gets written into `shortcuts.json`/`.winkrecipe` for frontmost-app pseudo-target shortcuts; `AppShortcut.frontmostTargetDisplayName` (localized) and `AppShortcut.displayAppName` (resolves the sentinel row to the localized label, passes every other shortcut's `appName` through unchanged) are display-only accessors layered on top. The same split applies to `WinkRecipeImportPlanner.PlannedShortcut.resolvedAppName` (stable) vs. `.displayAppName` (display). A localized string baked into persisted data goes stale the moment the user switches system language — same principle as issue #323's locale-stable persistence keys.
- Persistence/date-key formatters (issue #323) are locale-invariant by design and are untouched by this work — do not route them through the catalog.
